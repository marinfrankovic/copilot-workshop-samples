# Product Owner track — research & write great user stories

**Goal:** take a messy, real-world feature request and use Copilot to research
the ambiguity, then produce small, well-formed user stories with a solid,
testable Definition of Done.

## Files
- `feature-request.md` — the raw request as it arrived (your input).
- `user-story-template.md` — the story + Definition of Done shape to fill.

## Short demo (the loop)
1. **Research:** "Read `feature-request.md`. List the hidden assumptions and the questions I should ask before committing."
2. **Slice:** "Break this into 4–6 INVEST user stories. Use `user-story-template.md`."
3. **Write:** "For the top story, write Given/When/Then acceptance criteria, including edge cases."
4. **DoD:** "Draft a Definition of Done for a non-technical, mobile, auditable reporting feature."
5. **Refine:** "Prioritise with MoSCoW and tell me what to cut from release 1."

## Why this matters
The win isn't faster typing — it's **better thinking**: surfacing assumptions,
covering edge cases, and writing acceptance criteria you can actually test.

## Safety
- Copilot drafts; you own scope and priorities.
- Don't paste confidential customer data into prompts — the sample here is fictitious Contoso data.
